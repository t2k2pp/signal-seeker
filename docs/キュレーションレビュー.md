# キュレーションシステム レビュー結果

作成日: 2026-06-25

対象: SignalSeeker を「収集物を取捨選択し序列づけて届けるキュレーションシステム」として見たときの過不足。
確認範囲: `src/curation/score.ts` / `src/collector/attention.ts` / `src/report/model.ts` /
`src/summarizer/` / `src/db.ts` / `docs/設計書.md`。

## 結論

ランキングの土台（鮮度・注目度・内容の重み付き和、系列集約、ファクト無しの降格）は作り込まれている。
一方で「価値あるものを選び、重複や雑音を落とす」というキュレーション本来の選別力が弱い／欠けている箇所がある。
特に下記 ① ② は設計上の穴であり、最優先で塞ぐべき。

## 不足点（重要度順）

### ① `deepEndpoint`（重要記事の深掘り）が完全な未実装の死に設定

- 4チャンネル全部の `config.json` に書かれ、`types.ts`(`AppConfig.llm.deepEndpoint`) で型定義され、
  README・設計書にも載っているが、**コードのどこからも読まれていない**。
  `summarizer/index.ts` は `config.llm.endpoint` だけを使う。
- キュレーションの肝である「スコアが高い記事だけ上位モデルで掘り下げる」が動いていない。
  設計書 §14 でも「拡張候補」のまま。
- CLAUDE.md「フォールバック・場当たり対応の禁止」に照らすと、**使われない設定が4チャンネルに残っている状態**
  そのものが要対応。実装するか、設定・型・ドキュメントから外すか、どちらかに倒す。

### ② 同じ話題が複数ソースから来たときの重複排除がない

- 重複判定は DB の `(source_id, item_key)` 単位だけ（`db.ts`）。
  **別ソースが同じニュースを報じると、レポートに同じ話題が何度も並ぶ。**
- `groupBySeries` はバージョン番号（rc 等）の系列をまとめるだけで、「同一トピックの横断集約」はしない。
- 設計書 §14 は「題材の再出現頻度」を月次レポートの将来案として置いているが、
  **日次・週次の重複対策としては未着手**。キュレーション体験を最も下げる箇所。

### ③ 「重要度」の判定がキーワード正規表現に依存していて脆い

- `contentScore` / `isNoFacts` が日本語の `破壊的変更`・`ファクトなし` 等の正規表現マッチ頼み（`score.ts`）。
  LLM が言い回しを変えると**静かにスコアが崩れる**。
- 要約 LLM は自由テキストを返すだけで、重要度・カテゴリ・破壊的変更フラグなどの
  **構造化シグナルを出していない**。中核判断が「後段の文字列照合」になっているのは弱い。
- 改善方針: 要約時に JSON で `importance` や `breaking: true/false` を返させ、それをスコアに使う。

### ④ 足切り（しきい値）がなく、結局すべて配信される

- スコアは**並べ替えにしか使われず**、`demoteFactor` も順位を下げるだけ。
  `pendingItems()` の全件が配信対象。
- 「一定スコア未満は除外／付録送り」という `minScore` 的な仕組みがないため、
  **雑音を落とす**というキュレーションの片輪が欠けている。

### ⑤ 注目度シグナルが arXiv と GitHub リリースに偏っている

- `attention.ts` が拾えるのは HF upvote・引用数・GitHub star/reaction だけ。
  **RSS/HTML 記事（企業ブログ・ニュースなど大多数）は注目度 = 0** になり、スコアが鮮度＋内容だけで決まる。
- 結果としてランキングが arXiv/GitHub に有利に偏る。html 記事は `publishedAt` 欠落時に鮮度も一律 0.5 になりがち
  （`recencyScore`）で、**1ジャンル丸ごとスコアが平坦化**する。

### ⑥ 有用度のフィードバック・学習の経路がない

- 「この記事は役に立った／要らなかった」を記録して重みを調整する仕組みがない。完全に固定ヒューリスティック。
- Web クライアントがあるので、ここに評価入力を足すと ② ③ の改善材料にもなる（将来案）。

## できている点（評価）

- ToDo の項目（ログ／再開／DB 閲覧／設定メンテ／FTS5 全文検索）は実装済み。
- スコアの重み・係数を `config.curation` に集約しハードコードを避けている設計は良好。
- 系列集約・ファクト無しの末尾送り・dry 実行の繰越など、配信品質の工夫は丁寧。

## 優先度のまとめ

| 優先 | 項目 | 種別 |
| --- | --- | --- |
| 最優先 | ① deepEndpoint の死に設定 | 設計の穴（実装 or 撤去） |
| 最優先 | ② 横断重複排除なし | 機能欠落 |
| 高 | ③ 重要度判定がキーワード依存 | 堅牢性 |
| 中 | ④ 足切りしきい値なし | 機能欠落 |
| 中 | ⑤ 注目度シグナルの偏り | 公平性 |
